# Agent App — Backend Guidelines

Ce document décrit les conventions et la structure à respecter pour le développement backend d'une Agent App dans le toolkit v3.

> Ce fichier sert aussi de prompt agentique — il est référencé par les agents Copilot `/backend-builder` et `/quality`.

---

## First-run setup

1. **Copier `.env.example` → `back/.env`** et renseigner les deux variables obligatoires :
   ```bash
   AZURE_OPENAI_ENDPOINT=https://<votre-ressource>.openai.azure.com/
   AZURE_OPENAI_API_KEY=<votre-clé>          # laisser vide pour utiliser az login
   ```
   Ou, si vous utilisez l'authentification Azure CLI :
   ```bash
   az login
   # Laisser AZURE_OPENAI_API_KEY vide dans back/.env — DefaultAzureCredential prend le relais
   ```

2. **Installer les dépendances** et démarrer le serveur :
   ```bash
   cd back
   uv sync
   uv run uvicorn main:app --reload --port 8000
   ```

3. **Si Azure n'est pas configuré**, le framework intercepte l'erreur d'authentification et renvoie un message d'erreur SSE lisible à la place d'un HTTP 500.

## Pattern obligatoire : `DeclarativeAgentApp`

⚠️ **Les fonctions standalone `async def stream_xxx(...)` sont interdites.** Tout agent utilise `DeclarativeAgentApp`.

```python
from framework import DeclarativeAgentApp, TabSchema, StepContext, step, action
from framework.stream_safe import stream_safe
from services.llm_config import get_llm

from .schemas import MyInputTab, MyOutputTab
from .prompts import get_ui_message, get_step1_prompt

class MyApp(DeclarativeAgentApp):
    """One-sentence description of this agent app.

    Tabs:
        MyInputTab: User inputs.
        MyOutputTab: Generated results.
    """

    class Meta:
        tabs = [MyInputTab, MyOutputTab]

    @stream_safe
    @step(id="generate", persist_progress=True)
    async def generate(self, ctx: StepContext, inputs: MyInputTab, **kwargs):
        """Generate results from user input.

        Args:
            ctx: StepContext with username and language.
            inputs: Validated input tab.
            **kwargs: Additional flat state fields.

        Yields:
            SSEEvent dicts as text/event-stream data.
        """
        lang = inputs.lang or kwargs.get("lang", "fr")

        yield self.in_progress(step_id="generate", message=get_ui_message(lang, "init"), progress=5)

        llm = get_llm()
        prompt = get_step1_prompt(lang=lang, topic=inputs.topic)
        response = await llm.ainvoke([("system", prompt)])
        raw = response.content.strip()

        yield self.in_progress(step_id="generate", message=get_ui_message(lang, "parsing"), progress=80)

        result = _parse_result(raw, lang)

        yield self.completed(
            step_id="generate",
            message=get_ui_message(lang, "completed"),
            data=result.model_dump(),
        )
```

## Schemas (`schemas.py`)

```python
from pydantic import BaseModel, Field
from framework import TabSchema

# ─── Input tab ────────────────────────────────────────────────────────────────

class MyInputTab(TabSchema):
    """Input data provided by the user."""
    topic: str = Field("", description="Main subject")
    lang: str = Field("fr", description="Output language: 'fr' or 'en'")

# ─── Output tab ───────────────────────────────────────────────────────────────

class MyOutputTab(TabSchema):
    """Results generated by the agent steps."""
    result: str = Field("", description="Generated result from Step 1")
    items: list[str] = Field(default_factory=list, description="List of items")

# ─── Typed output models (used in completed() calls) ─────────────────────────

class Step1Output(BaseModel):
    """Typed output for the 'generate' step."""
    result: str = Field(description="Generated result")
    items: list[str] = Field(default_factory=list)
```

### Règles `TabSchema`

- **Tous les champs ont des valeurs par défaut** — jamais de champ requis sans défaut
- `str` → `Field("")`, `list` → `Field(default_factory=list)`, `bool` → `Field(False)`
- Noms de champs **uniques** sur tous les tabs d'un même agent (état = flat dict)

## Helpers SSE

```python
# Progression intermédiaire
yield self.in_progress(
    step_id="generate",       # ← identifiant du step (obligatoire)
    message="En cours...",    # ← message affiché à l'utilisateur
    progress=50,              # ← 0-100
)

# Fin avec succès + données métier
yield self.completed(
    step_id="generate",
    message="Terminé !",
    data={"result": "...", "items": [...]},   # ← persisté dans l'état
)

# Erreur
yield self.error(
    step_id="generate",
    message="Une erreur est survenue. Veuillez réessayer.",
)
```

Les données passées dans `completed(data=...)` sont fusionnées dans le flat state et disponibles pour les steps suivants via `**kwargs`.

## Convention de shape de l'`app_state` (CRITIQUE pour le frontend)

Le framework écrit chaque clé du `data=` **directement à plat** dans l'`app_state` persisté (pas de wrapper implicite). C'est ce qui sera renvoyé par `GET /agent-apps/{app_id}/state` et que le store Zustand frontend va projeter dans `loadFromBackend()`.

### Règle 1 — Toujours nommer explicitement les clés métier

```python
# ✅ BIEN : clés explicites, faciles à projeter côté front
yield self.completed(
    step_id="generate_personas",
    data={
        "persona1": persona1.model_dump(),
        "persona2": persona2.model_dump(),
    },
)

# ❌ MAL : enveloppe générique, le front doit deviner ce qu'il y a dedans
yield self.completed(step_id="generate_personas", data={"result": {...}})
```

L'`app_state` après ce step contiendra :

```json
{
  "persona1": { ... },
  "persona2": { ... },
  "_status": "completed",
  "_last_step_id": "generate_personas"
}
```

### Règle 2 — `snake_case` pour toutes les clés persistées

Les clés doivent être en `snake_case` (Python convention). Le store frontend les projette ensuite vers du `camelCase` :

```python
# Backend
yield self.completed(data={"key_points": [...], "next_steps": [...]})
```

```typescript
// Frontend
const keyPoints = Array.isArray(flat.key_points) ? flat.key_points : [];
const nextSteps = Array.isArray(flat.next_steps) ? flat.next_steps : [];
```

### Règle 3 — Préfixe `_` réservé au framework

Le framework écrit lui-même `_status`, `_last_step_id` (et autres champs internes) à plat. Ne jamais utiliser de clé commençant par `_` dans votre `data=`.

### Règle 4 — Outputs sérialisables JSON uniquement

Le state est persisté en JSON (Mongo en prod, fichier en local). Tout ce qui est dans `data=` doit être JSON-serialisable :

```python
# ✅ Pydantic models → toujours .model_dump()
data={"persona": persona.model_dump()}

# ✅ Listes / dicts / primitives
data={"messages": [{"role": "user", "content": "..."}]}

# ❌ Objets non-sérialisables (datetime brut, BinaryIO, etc.)
data={"timestamp": datetime.now()}  # → utiliser .isoformat()
```

### Pourquoi cette convention compte

Le frontend a une discipline stricte (voir `AGENT_APP_GUIDELINES_FRONT.md` → "Persistance d'état") :

- Le **localStorage** ne contient QUE les inputs et la navigation
- Tous les **outputs générés** sont rechargés depuis le backend au montage via `loadFromBackend()`
- Le store reconstitue les outputs en projetant chaque clé attendue depuis le state plat

Si vous wrappez vos outputs dans une enveloppe `{"result": {...}}` ou si vous changez le nom des clés entre versions, vous cassez la projection frontend et l'utilisateur perd ses résultats au refresh. **Tenez le contrat de shape stable.**

## Décorateurs

### `@step(id=..., persist_progress=...)`

```python
@stream_safe
@step(id="generate", persist_progress=True)
async def generate(self, ctx: StepContext, inputs: MyInputTab, **kwargs):
    ...
```

- `id` : step_id utilisé dans les routes SSE et les events (snake_case)
- `persist_progress=True` : persiste l'état en BDD (dans le monorepo Neo) — mettre `True` pour les steps principaux
- Le framework injecte automatiquement le bon tab dans `inputs` selon l'ordre des tabs dans `Meta`

### `@action(id=..., reads=[...])`

Pour les mutations légères (modifier un seul champ sans restreamer tout le step) :

```python
@stream_safe
@action(id="update_field", reads=[MyOutputTab])
async def update_field(self, ctx: StepContext, inputs: MyOutputTab, **kwargs):
    """Update a single field in the output tab.

    Args:
        ctx: StepContext with username and language.
        inputs: Validated output tab.
        **kwargs: Additional flat state fields.

    Yields:
        SSEEvent dict with updated field.
    """
    new_value = _compute(inputs.result)
    yield self.completed(step_id="update_field", data={"result": new_value})
```

### `@stream_safe`

**Obligatoire sur chaque `@step` et `@action`.**
Intercepte toutes les exceptions et yield un message d'erreur compréhensible en français.

```python
@stream_safe   # ← toujours en premier (avant @step)
@step(id="generate")
async def generate(self, ...):
```

## Prompts (`prompts.py`)

Ne jamais hardcoder des prompts dans `app.py`. Centraliser dans `prompts.py` :

```python
# prompts.py

_STEP1_SYSTEM_FR = """
Tu es un expert en {domain}.
Sujet : {topic}
Génère un résultat structuré en JSON :
{{"result": "...", "items": ["...", "..."]}}
"""

_STEP1_SYSTEM_EN = """
You are an expert in {domain}.
Topic: {topic}
Generate a structured JSON result:
{{"result": "...", "items": ["...", "..."]}}
"""

_UI_MESSAGES_FR = {
    "init": "Initialisation...",
    "generating": "Génération en cours...",
    "parsing": "Analyse des résultats...",
    "completed": "Génération terminée !",
}

_UI_MESSAGES_EN = {
    "init": "Initializing...",
    "generating": "Generating...",
    "parsing": "Parsing results...",
    "completed": "Generation complete!",
}


def get_step1_prompt(lang: str, topic: str, domain: str = "") -> str:
    """Build the system prompt for step 1.

    Args:
        lang: Output language ('fr' or 'en').
        topic: User's topic.
        domain: Optional domain context.

    Returns:
        Formatted system prompt string.
    """
    template = _STEP1_SYSTEM_FR if lang.startswith("fr") else _STEP1_SYSTEM_EN
    return template.format(topic=topic, domain=domain)


def get_ui_message(lang: str, key: str) -> str:
    """Get a UI message by key for the given language.

    Args:
        lang: Output language ('fr' or 'en').
        key: Message key (e.g., 'init', 'generating', 'completed').

    Returns:
        Localized UI message string.
    """
    messages = _UI_MESSAGES_FR if lang.startswith("fr") else _UI_MESSAGES_EN
    return messages.get(key, key)
```

## Gestion des erreurs

Ne jamais utiliser de `try/except` manuels dans les steps — `@stream_safe` le fait automatiquement. Pour une erreur métier prévisible, utiliser `self.error()` :

```python
if not inputs.topic.strip():
    yield self.error(
        step_id="generate",
        message="Veuillez saisir un sujet avant de lancer la génération."
        if lang.startswith("fr")
        else "Please enter a topic before starting.",
    )
    return
```

Le service `services/process_files.py` extrait et résume des documents.

### Fonction principale

```python
from services.process_files import print_or_summarize

document_context = ""
if file_paths:
    parts = []
```

## Traitement de documents

Le service `services/process_files.py` fournit une extraction et une condensation de documents locaux. Il est le miroir direct de la fonction `print_or_summarize` du backend Neo principal, sans dépendance Azure Blob Storage.

### Règle d'or : fichiers par référence, jamais par valeur

> **JAMAIS de `bytes`, de base64, ou de contenu binaire en clair dans un champ de Tab.** Un fichier circule toujours sous forme de **référence** (chemin absolu en local, URL Blob Storage en prod).

Dans le toolkit, la référence est le `path` retourné par `POST /files/upload` (ex: `/.../tempfiles/<uuid>_mon_doc.pdf`). En production Elio, le même champ contient une URL Azure Blob Storage. **Le contrat reste le même des deux côtés** : votre code agent ne reçoit qu'une chaîne, qu'il passe à `print_or_summarize` (qui sait gérer les deux).

### Déclarer un champ binaire dans un Tab : `is_blob=True`

Quand un champ d'un Tab transporte un fichier (ou une liste de fichiers), il **doit** être marqué comme blob via `json_schema_extra` :

```python
from pydantic import Field
from framework.base_agent_app import TabSchema

class MyInputTab(TabSchema):
    topic: str = Field(default="", description="Sujet à traiter")
    # Liste de références fichier (chemin local OU URL Blob).
    file_paths: list[str] = Field(
        default_factory=list,
        description="Chemins/URLs des fichiers uploades",
        json_schema_extra={"is_blob": True},   # ← OBLIGATOIRE
    )
```

Pourquoi `is_blob` compte :

- Le manifest exposé par `get_manifest()` annonce le champ comme blob, ce que le frontend Elio (et un futur orchestrateur de handover) utilise pour router le fichier vers le bon stockage.
- C'est aussi un garde-fou conceptuel : si vous êtes tenté d'y mettre du base64, le flag vous rappelle que ce champ est une référence, pas un contenu.

### Pourquoi pas de bytes ni de base64

| Problème                         | Conséquence                                                                                           |
| -------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Payload SSE                      | base64 multiplie la taille ×1.33, sature le canal SSE et le navigateur                                |
| `app_state` Mongo (prod)         | Document limité à 16 Mo — un seul PDF moyen le fait exploser                                          |
| Reload front (`loadFromBackend`) | Les outputs sont projetés depuis le state plat ; un blob inline serait re-téléchargé à chaque refresh |
| Handover entre agents            | Une référence est portable d'un agent à l'autre, un blob inline non                                   |
| localStorage front               | Limite ~5 Mo — un base64 fait sauter le quota et casse le store                                       |

### Lecture du fichier dans un step

Les steps reçoivent simplement la chaîne (chemin local dans le toolkit, URL en prod) et la passent au service de lecture :

```python
from services.process_files import print_or_summarize

@stream_safe
@step(id="analyze", persist_progress=True)
async def analyze(self, ctx: StepContext, inputs: MyInputTab, **kwargs):
    document_context = ""
    for path in inputs.file_paths:        # ← chemins/URLs, jamais des bytes
        content = await print_or_summarize(path, language=ctx.language)
        document_context += f"\n\n## {path}\n{content}"
    # ... appel LLM avec document_context
```

Côté toolkit, `print_or_summarize` lit le disque local sous `tempfiles/`. Côté prod Neo, la même fonction télécharge depuis Azure Blob Storage. Votre code agent ne change pas.

### Récapitulatif

- Tab field qui transporte un fichier → `list[str]` (ou `str`) avec `json_schema_extra={"is_blob": True}`.
- Le payload SSE entrant ne contient que des références (chemins ou URLs), jamais de bytes ni de base64.
- L'`app_state` persisté ne contient que des références — reload et handover restent légers.
- Le toolkit simule le Blob Storage par un dossier `tempfiles/` ; le contrat HTTP/Pydantic est identique à la prod.

| Extension                             | Librairie                                         | Notes                                       |
| ------------------------------------- | ------------------------------------------------- | ------------------------------------------- |
| `.pdf`                                | PyMuPDF + Azure Document Intelligence (optionnel) | DI utilisé en fallback si le PDF est scanné |
| `.docx`                               | python-docx                                       |                                             |

### Fonction principale : `print_or_summarize`

```python
from services.process_files import print_or_summarize

document_context = ""
if file_paths:
    parts = []
    for path in file_paths:
        try:
            content = await print_or_summarize(path, language="FR")
            parts.append(content)
        except Exception as exc:
            logger.warning(f"Could not process {path}: {exc}")
    document_context = "\n\n".join(parts)
```

**Comportement** :

- Si le contenu extrait est **≤ 10 000 caractères** → retourné tel quel (préfixé avec le nom du fichier).
- Si le contenu est **> 10 000 caractères** → résumé automatiquement via le LLM configuré.

**Signature complète :**

```python
await print_or_summarize(
    file_path: str,
    threshold: int = 10_000,   # seuil de résumé en caractères
    language: Literal["FR", "EN"] = "FR",
    model: str = "",           # laisser vide pour utiliser le modèle par défaut
)
```

### Injection dans le prompt

Une fois le contexte construit, l'injecter dans le prompt LLM via le placeholder `{document_context}` :

```python
MY_PROMPT = """
You are an expert analyst.
{document_context}

Based on these documents, do X...
"""

prompt = MY_PROMPT.format(
    document_context=(
        f"The user has provided the following documents:\n{document_context}"
        if document_context
        else "No documents provided."
    ),
    # ... autres variables
)
```

### Signature de l'étape avec support de fichiers

Ajouter `file_paths: list[str] | None = None` comme paramètre optionnel :

```python
async def stream_my_step(
    username: str,
    topic: str,
    file_paths: list[str] | None = None,  # chemins absolus retournés par /files/upload
    **kwargs: Any,
) -> AsyncGenerator[dict[str, Any], None]:
    document_context = ""
    if file_paths:
        parts = []
        for path in file_paths:
            try:
                parts.append(await print_or_summarize(path))
            except Exception as exc:
                logger.warning(f"Could not process {path}: {exc}")
        document_context = "\n\n".join(parts)
    # ... suite du traitement
```

### Azure Document Intelligence (optionnel)

Pour les PDFs scannés (images uniquement, sans couche texte) :

1. Configurer `AZURE_DOCUMENT_INTELLIGENCE_ENDPOINT` dans `.env`
2. Avoir le rôle RBAC `Cognitive Services User` sur la ressource
3. Le service bascule automatiquement sur Document Intelligence si PyMuPDF extrait < 4 000 caractères

Variables d'environnement :

```bash
# Bascule automatique (recommandé)
AZURE_DOCUMENT_INTELLIGENCE_ENDPOINT=https://your-resource.cognitiveservices.azure.com/

# Forcer Document Intelligence pour tous les PDFs
FORCE_DOC_INTELLIGENCE=true
```

## Transcription audio

Le service `process_files.py` prend aussi en charge les fichiers audio. La transcription est réalisée via **Azure OpenAI Whisper**.

### Formats supportés

| Extension | Format     |
| --------- | ---------- |
| `.mp3`    | MPEG Audio |
| `.m4a`    | AAC        |
| `.wav`    | WAV/PCM    |
| `.ogg`    | Ogg Vorbis |
| `.webm`   | WebM Audio |

### Configuration

1. Déployer un modèle Whisper sur votre ressource Azure OpenAI
2. Configurer `WHISPER_DEPLOYMENT_NAME` dans `.env` (par défaut : `"whisper"`)
3. Le dispatch est automatique : tout fichier audio passe par `_extract_audio()`, qui retourne la transcription texte

```python
# Dans votre étape, rien de spécial — print_or_summarize gère les audio comme les documents
content = await print_or_summarize(file_path)
```

## Génération de documents

Le service `services/generate_files.py` permet de générer des fichiers DOCX et PPTX à partir de contenu LLM. Deux modes :

### Mode libre (free-form)

Créer des documents à partir de contenu brut :

```python
from services.generate_files import markdown_to_docx, text_to_docx, slides_to_pptx

# Markdown → DOCX
path = markdown_to_docx(markdown_text, title="Mon rapport")

# Texte brut → DOCX
path = text_to_docx(plain_text, title="Notes de réunion")

# Liste de slides → PPTX
slides = [{"title": "Intro", "content": "Contenu"}, ...]
path = slides_to_pptx(slides, title="Ma présentation")
```

### Mode template

Remplir des templates existants avec des placeholders.

Deux templates FR A4 branded sont inclus par défaut dans `back/templates/` :

| Constante                | Fichier                                       | Usage                          |
| ------------------------ | --------------------------------------------- | ------------------------------ |
| `DOCX_TEMPLATE_OP`       | `FR_WORD TEMPLATE_OP_A4_2025.docx`            | Template opérationnel (défaut) |
| `DOCX_TEMPLATE_WE_BLANK` | `FR_WORD TEMPLATE_WE_A4_2025_BLANK PAGE.docx` | Template page blanche          |

```python
from services.generate_files import (
    fill_docx_template,
    fill_pptx_template,
    DOCX_TEMPLATE_OP,
    DOCX_TEMPLATE_WE_BLANK,
)

# DOCX : remplace les {{clé}} — template par défaut (OP) si template_path omis
path = fill_docx_template(
    placeholders={"titre": "Mon rapport", "contenu": "..."},
    output_name="rapport_final",
    # template_path=DOCX_TEMPLATE_WE_BLANK,  # changer si besoin
)

# PPTX : remplace les <<clé>> dans le template
path = fill_pptx_template(
    template_path="templates/pres.pptx",
    placeholders={"titre": "Ma pres", "bullet1": "..."},
    output_name="pres_finale",
)
```

Tous les fichiers générés sont sauvegardés dans `tempfiles/` avec un préfixe UUID et peuvent être téléchargés via `GET /files/{filename}/download`.

## Prompts bilingues (OBLIGATOIRE)

Les prompts LLM **doivent** être séparés du code métier et organisés dans deux fichiers :

```
back/agents/your_agent/
├── prompt_fr.py      # Prompts en français
├── prompt_en.py      # Prompts en anglais
├── step1_xxx.py      # Code métier — importe les prompts
└── step2_xxx.py
```

### Structure des fichiers de prompts

Chaque fichier contient des **constantes** qui sont des templates de prompt avec des `{placeholders}` :

```python
# prompt_fr.py
PERSONA_GENERATION_PROMPT = """Tu es un créateur de personnages créatif.
Sujet : {topic}
{document_context}
..."""

DISCUSSION_SYSTEM_PROMPT = """Tu es un simulateur de conversation.
..."""
```

```python
# prompt_en.py
PERSONA_GENERATION_PROMPT = """You are a creative character designer.
Topic: {topic}
{document_context}
..."""

DISCUSSION_SYSTEM_PROMPT = """You are a conversation simulator.
..."""
```

### Sélection dynamique dans les étapes

Le paramètre `interface_language` (envoyé par le frontend) détermine le fichier de prompts utilisé :

```python
from agents.your_agent import prompt_en, prompt_fr


def _get_prompts(interface_language: str):
    if interface_language.startswith("fr"):
        return prompt_fr
    return prompt_en


async def stream_my_step(
    username: str,
    topic: str,
    interface_language: str = "fr",  # ← ajouté
    **kwargs: Any,
) -> AsyncGenerator[dict[str, Any], None]:
    prompts = _get_prompts(interface_language)
    prompt = prompts.MY_PROMPT.format(topic=topic, ...)
    # ... suite du traitement
```

> **Important** : `interface_language` est transmis automatiquement par le frontend dans le payload SSE. Toute nouvelle étape **doit** accepter ce paramètre.

## Étapes éditables (OBLIGATOIRE)

Chaque étape doit être conçue pour que ses résultats soient **modifiables** par l'utilisateur avant de passer à l'étape suivante. Les données générées par le LLM sont un point de départ, pas un résultat final.

Concrètement :

- Les résultats intermédiaires (personas, paramètres, textes, etc.) doivent être stockés dans le store Zustand côté frontend
- Le frontend doit proposer des champs éditables pour chaque donnée générée
- L'étape suivante utilise les données **modifiées par l'utilisateur**, pas les données brutes du LLM

```python
# Le backend retourne des données structurées que le frontend pourra éditer
yield {
    "step": "completed",
    "status": "completed",
    "progress": 100,
    "personas": personas_data,  # ← données éditables par l'utilisateur
}
```
